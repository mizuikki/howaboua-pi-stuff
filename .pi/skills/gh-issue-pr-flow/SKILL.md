# GH Issue PR Flow — Howaboua Pi Stuff

Use this instead of the global generic PR flow in this repository.

## Repo facts

- Monorepo root: `/home/igorw/Work/howaboua-pi-stuff`
- Package manager: Bun workspaces.
- Workspace packages: `packages/*`.
- Release tooling: Changesets.
- Aggregates:
  - `@howaboua/pi-stuff` — everything
  - `@howaboua/pi-extensions` — extension packages
  - `@howaboua/pi-skills` — skill packages
- Normal packages stay separately installable.
- Aggregate package bumps are auto-generated by `bun run changeset:aggregates` from pending changesets.
- Local hooks live in `.githooks/`; install with `bun run hooks:install` if needed.

## Hard rules

- Do not manually bump package versions for normal feature/fix PRs. Use `bun changeset`.
- Do not manually edit aggregate package versions. The release workflow handles them.
- Do not manually add changesets for `@howaboua/pi-stuff`, `@howaboua/pi-extensions`, or `@howaboua/pi-skills` unless the aggregate package itself changed.
- Keep changes scoped to the relevant `packages/<name>` directory plus any required repo-level workflow/docs/scripts.
- Do not run all package checks by default. Prefer changed-package validation.
- For `pi-codex-conversion`, do not rebuild or commit apply_patch binaries unless apply-patch source/wrapper paths changed or the user asked.
- Before GitHub side effects, keep the global sponsor rule: ensure `.github/FUNDING.yml` exists or recommend/add `github: IgorWarzocha`.

## Common commands

```bash
bun install
bun run check:changed
bun run changeset:check
bun run changeset:aggregates
bun changeset
```

Changed-package validation:

```bash
bun run check:changed
```

Full validation only when appropriate:

```bash
bun run check
```

Bypass local hooks only for WIP/emergency work:

```bash
SKIP_CHECK_CHANGED=1 git commit -m "wip"
SKIP_CHANGESET_CHECK=1 git push
SKIP_HOOKS=1 git commit -m "wip"
```

## Workflow for implementing a package change

1. Inspect state:
   ```bash
   git status --short --branch
   ```
2. Create or switch to a focused branch from `main` unless the user says otherwise.
3. Make scoped edits.
4. Run:
   ```bash
   bun run check:changed
   ```
5. If the change should ship, create a changeset:
   ```bash
   bun changeset
   ```
   - Select only the directly changed individual package(s).
   - Usually choose `patch`.
   - Use `minor` for user-visible new capability.
   - Use `major` only for breaking changes.
6. Do not create aggregate changesets manually. The release workflow runs:
   ```bash
   bun run changeset:aggregates
   ```
7. Commit all relevant files, including the `.changeset/*.md` file.

## Workflow for docs/tooling-only changes

- If no package behavior changes, no changeset is required.
- Examples that usually do **not** need a changeset:
  - README-only repo guidance
  - CI workflow edits
  - git hook/script improvements that are not shipped in an npm package
- Still run targeted validation, usually:
  ```bash
  bun run check:changed
  ```
  or script-specific smoke commands.

## PR body template

Use a concise PR body:

```markdown
## Summary
- ...

## Validation
- `bun run check:changed`

## Release
- Changeset: yes/no
- Packages: `@howaboua/...`
- Aggregates: auto-bumped by `bun run changeset:aggregates`

## Sponsor button
- present / added / inherited / missing; follow-up recommended
```

If linked to an issue, use `Closes #123` only when fully resolved, otherwise `Refs #123`.

## Release behavior to remember

When a package changes and has a changeset, the release workflow:

1. Runs `bun run changeset:aggregates`.
2. Adds a generated aggregate changeset when needed.
3. Lets Changesets open/update the Version Packages PR.
4. Publishes changed individual package(s) and relevant aggregate package(s) after that PR merges.

Example: changing `@howaboua/pi-vent` with a patch changeset publishes:

- `@howaboua/pi-vent`
- `@howaboua/pi-extensions`
- `@howaboua/pi-stuff`

## apply_patch binary workflow

`packages/pi-codex-conversion` contains the Codex-style `apply_patch` tool and bundled binaries.

Only trigger the expensive binary workflow when these change:

- `packages/pi-codex-conversion/vendor/apply-patch-src/**`
- `packages/pi-codex-conversion/scripts/build-apply-patch-binary.mjs`
- `packages/pi-codex-conversion/scripts/sync-apply-patch-source.mjs`
- `packages/pi-codex-conversion/bin/apply_patch`
- `packages/pi-codex-conversion/bin/apply_patch.cmd`
- `packages/pi-codex-conversion/src/tools/apply-patch-binary.ts`

For normal TypeScript changes in `pi-codex-conversion`, reuse the existing bundled binaries.

## Asking Codex for review

When opening a PR, post the standard review request unless the user says not to:

```text
@codex please review this PR and give me 10-20 issues if any. Categorize findings as required, recommended, or optional.
```

Do not re-post on every update unless explicitly asked.

## Final response checklist

- Branch and working tree state known.
- Changed packages listed.
- Validation command and result listed.
- Changeset status stated.
- PR/issue links included if created.
- Sponsor button status stated for GitHub side effects.
